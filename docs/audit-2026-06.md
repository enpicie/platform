# Platform Audit — 2026-06-21

Audit of the `enpicie` personal platform against the stated goal:

> *Be my own platform team. Reusable pieces → low cognitive load → start new projects fast without sacrificing quality.*

Measured against three success criteria:

1. **Onboard without meetings** — infra + deploy are understandable without reading disjoint parts.
2. **Deploy new projects without significant setup.**
3. **Low maintenance** — no babysitting pipelines, no constant test rewrites.

Scope reviewed: 24 org repos, the Notion Core Context spec library (General Platform / Coding / API / Design / Mobile + CI/CD spec + stack/app specs), and the actual CI/Terraform layout of the shipped apps `find-my-fgc`, `adomi-san-bot`, and `fgc-league-sheets`.

> Note: `dia` is an experimental rebuild (far from shipped) and is **excluded** from this audit. The seeding tool is a planned full-stack app.

---

## Verdict

**The platform is genuinely strong at the unit level and weak at the seams.** The reusable pieces are well-factored, versioned, and secure. What's missing is the connective tissue that turns "a great set of parts" into "a platform a second person can use" — the three things you named as the product: **centralized docs, a plug-and-play starter pipeline, and one config convention that never changes.**

The single highest-leverage move is **shipping the env/secrets single source of truth** (the keystone). It's the convention you want in every project, and it's currently the *least* consistent part of the platform — `adomi-san-bot` even routes some secret values through Terraform state. Standardizing this once pays off in every project after.

---

## ✅ Standards already met (keep these)

- **Clean taxonomy.** `gh-action-*` (CI units), `tf-module-*` (infra units), `aws-*` (core infra) is a clear, scalable naming system. A newcomer can guess what a repo is from its name.
- **Reusable pieces are versioned and pinned.** `terraform-run@v1.0.0`, `s3-upload@v1.0.1`, `tf-module-s3-cloudfront-website@v1.2.0`, `tf-module-ecs-alb-service@v1.2.0`. The CI/CD spec mandates version pinning (never `@main`) and the modules deliver semver tags. This is the hard part of platform work and it's done.
- **Strong security posture.** GitHub OIDC (no long-lived keys), IAM roles scoped per resource group, one Terraform state file per component (`projects/{app}/{env}/{component}.tfstate`) so each run uses minimum permissions. Empty S3 backend blocks injected by the composite action. This is better than most professional setups.
- **Mature, composable standards** governing how projects are built (platform/coding/API/design/CI-CD), composed on init into a derived `CLAUDE.md` per project. (These currently live in your private notes; R2 brings the shared parts into this repo so they stand alone.)
- **Clean migration hygiene.** HCP Terraform → OpenTofu + S3 migration is complete; legacy `action-workflow-hcp-*` repos archived rather than deleted.
- **A faithful reference implementation exists.** `find-my-fgc` matches the CI/CD spec (`deployment.env` + `release.yml` + `workflow-build.yml` + `workflow-deploy.yml`, `terraform/{frontend,backend,infra}`).

## ⚠️ Partially met — specific gaps

### G1 — CI structure has drifted across apps, and the spec covers only one archetype (hurts goal #1, #3)
The spec's central promise is *one* three-file workflow pattern. Across the shipped apps, only one follows it:

| App | `deployment.env` | `release.yml` | build/deploy split | Notes |
|---|---|---|---|---|
| find-my-fgc | ✅ | ✅ | ✅ | Reference impl; also adds `bootstrap.yml`, `workflow-bootstrap.yml` |
| adomi-san-bot | ❌ | ✅ | ❌ (deploy only) | `development.yml`, `workflow-register-commands.yml`, `workflow-run-unit-tests.yml` |
| fgc-league-sheets | n/a | ❌ | n/a | **Different archetype entirely** — Google Apps Script via `clasp push` (`deploy-to-sheet.yml`), no AWS/Terraform. The CI/CD spec doesn't describe this path at all. |

**Impact:** onboarding to any non-reference app requires a per-repo explanation — the opposite of goal #1. The spec describes a pattern only `find-my-fgc` follows, and says nothing about real archetypes you already ship (bot/scheduled, Apps Script extension).
**Why it happened:** the apps are legitimately different archetypes. The spec assumes one shape; reality already has several. The fix isn't to force conformance — it's to *name a finite set of archetypes* and give each a blessed pattern (R3).

### G2 — No single source of truth for env vars + secrets; secret values can leak into Terraform state (hurts goal #1, #3) — **the keystone gap**
This is the convention you named as the core that must persist in every project, and today it's the least consistent thing in the platform. `adomi-san-bot` shows the reality — two contradictory patterns in one `secrets.tf`:
- `startgg_api_token` / `discord_bot_token`: Terraform **creates the secret value** from a `var.*`, so the value flows GitHub Actions → tfvars → **Terraform state** (a leak surface).
- `sheets_credentials`: imported as a pre-populated data source (value never in state — the correct pattern).

There's no single declaration of "what config does this app need." Names are ad-hoc per resource (though already single-sourced from `config.env` — good instinct). Local vs prod config can drift with nothing to catch it. The good parts (runtime fetch, scoped IAM) exist but aren't standardized.
**Fix:** a manifest-driven model — `config/app-config.yaml` as the SSOT, AWS Secrets Manager for values, Terraform owning the container + IAM only. See [docs/env-and-secrets.md](./docs/env-and-secrets.md) and [ADR-003](./docs/decisions/ADR-003-env-secrets-single-source-of-truth.md) (R1).

### G3 — No centralized, self-contained docs in GitHub (the "disjoint parts" you named) (hurts goal #1)
Shared platform knowledge that *does* live in GitHub is split across the `Infrastructure-Documentation` repo (good — architecture diagram, OIDC, migration), per-repo READMEs, and per-repo `CLAUDE.md`. There is **no single GitHub-visible "start here" that someone can follow end to end** without your private notes or toolchain. This repo (`enpicie-platform`) is the fix and is the documentation home going forward; `Infrastructure-Documentation` folds into it over time.

### G4 — Minor cruft / inconsistency (low impact, easy wins)
- Region drift across AWS apps (e.g. `find-my-fgc` / spec example use `us-east-2`); pick one default and note exceptions per project.
- The CI/CD spec link points to `action-workflow-build-python-lambda-layer-zip` but the repo is `gh-action-workflow-build-python-lambda-layer-zip`.
- An open `TODO` in Infrastructure-Documentation: move `adomi-san-bot` Lambda-zip handling fully into Terraform.

*(Earlier draft flagged unversioned `aws-*` repos — removed. These are intentional singletons at personal-org scale that just need to exist, not version-pinned products.)*

## ❌ Not met — missing capabilities

### G5 — No portable, plug-and-play starter (hard blocker for goal #1, #2 for *other people*)
24 repos, **none** is a runnable template or generator. Today, starting a project depends on your private toolchain and tacit knowledge rather than a `git clone template && configure && tag` path. The product you described — *a reusable/plug-and-play starting pipeline that can be adjusted per project* — doesn't exist yet as an artifact. Goal #1 ("onboard someone to make changes") is currently met only for an owner-equivalent. The starter should be flexible (copy + adjust), not a rigid generator.

### G6 — No deployment-target decision for the new JS apps (strategic gap, goal #3)
The platform is AWS-by-default (S3+CloudFront frontend, Lambda/APIGW or ECS/ALB backend). The next full-stack TS app (the seeding tool, and eventually the `dia` rebuild once it stabilizes) faces a choice with no written rule behind it: *"AWS+Terraform vs Vercel vs Railway, and when?"* You've explored Vercel/Railway, but the decision isn't encoded, so each new app either re-litigates it or silently diverges. The AWS path's strengths (IAM scoping, state management) are overhead a push-to-deploy web app doesn't need. Encoding this is the biggest single lever on "low maintenance" for new work.

| **R4** | Define **one canonical CI pattern per archetype** (`static-frontend`, `container-backend`, `scheduled-job`, `full-stack-js`, `bot`, `apps-script`) and one reference repo each. Bless variants explicitly — finite, named, no snowflakes. | G1 | M |
